Guia de Estudo — playwright-e2e-ts

Playwright TypeScript Node.js 20 Referência completa pra entender cada pedaço do projeto e dominar a stack de testes com Playwright + TypeScript.

1. Visão geral

Este repositório é a versão Node.js/TypeScript do projeto playwright-e2e-testing (escrito em Python). Mesmos alvos, mesmos padrões de engenharia (Page Object Model, fixtures, CI), em duas stacks diferentes.

O que tem dentro

  • 25 testes passando em ~22s localmente.
  • 5 Page Objects tipados (Login, Dropdown, Checkboxes, DynamicLoading, JavaScriptAlerts).
  • Fixtures customizadas via test.extend<>() — o equivalente TS do conftest.py.
  • Testes de API com request fixture, sem libs extras.
  • CI com GitHub Actions rodando Chromium headless.
  • Documentação: README.md + docs/python-to-typescript.md (tabela comparativa).

Alvo dos testes

  • UI: the-internet.herokuapp.com — site público didático com fluxos isolados.
  • API: httpbin.org — API pública que ecoa requests (perfeita pra validar comportamento sem mock).

Destaques técnicos

  • Stack moderna (Node 20 + @playwright/test + TypeScript strict).
  • Tabela Python → TypeScript mostrando a conversão entre as duas stacks.
  • API + E2E no mesmo runner, sem libs extras.
  • Config comentada e justificada (fullyParallel, trace: 'on-first-retry', retries: CI ? 1 : 0).

Fluxo de uma execução de teste

  1. npm test chama npx playwright test.
  2. Playwright lê playwright.config.ts → sabe que os testes estão em ./tests, baseURL é o-internet, projeto é chromium.
  3. Pra cada arquivo *.spec.ts, Playwright lê os imports — o test importado de fixtures/pages.ts estende o test base com as 5 pages como fixtures.
  4. Quando um teste declara async ({ loginPage, page }) => ..., Playwright:
    1. Cria um BrowserContext novo (cookies limpos).
    2. Cria um Page novo.
    3. Constrói new LoginPage(page) e injeta como loginPage.
  5. O teste roda. Ao terminar, o contexto é descartado (isolamento total).
  6. Falhou? Playwright tira screenshot, grava vídeo (se configurado) e — se retries > 0 — retesta com trace ativo.

3. playwright.config.ts — linha a linha

Esse é o primeiro arquivo a entender — define como toda a suíte se comporta. Saber por que cada linha está lá é essencial.

import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',

  forbidOnly: !!process.env.CI,           // bloqueia test.only no CI
  fullyParallel: true,                    // paraleliza testes dentro de um spec
  retries: process.env.CI ? 1 : 0,        // 1 retry no CI, 0 local
  workers: process.env.CI ? 2 : undefined, // 2 workers no CI, auto local

  reporter: process.env.CI
    ? [['list'], ['html', { open: 'never' }]]
    : [['html', { open: 'never' }]],

  use: {
    baseURL: 'https://the-internet.herokuapp.com',
    trace: 'on-first-retry',    // só grava trace quando retesta
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
    actionTimeout: 10_000,
    navigationTimeout: 15_000,
  },

  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
  ],
});

Decisões justificadas

fullyParallel: true

Cada teste dentro de um .spec.ts roda em paralelo (num worker separado). Acelera a suite. Só funciona porque escrevi os testes independentes (sem ordem obrigatória). Em projetos com testes acoplados, ligar isso quebra tudo.

retries: process.env.CI ? 1 : 0

Local eu quero ver a falha imediatamente pra consertar. No CI, 1 retry pega flakes de rede (ex.: herokuapp.com demorou 3s pra responder uma vez). Se falhar 2x, provavelmente é bug real. Nunca passaria de retries: 1 — retries demais escondem bugs.

workers: process.env.CI ? 2 : undefined

undefined = Playwright usa metade dos cores disponíveis. Local isso é rápido. No CI (GitHub Actions runner com 2 vCPUs), setar explicitamente 2 é mais previsível que deixar automático.

trace: 'on-first-retry'

Trace é caro (arquivo de dezenas de MB com timeline completa, DOM snapshots, console, network). Gerar em todo teste é desperdício. 'on-first-retry' só gera quando o teste já falhou uma vez — quando você realmente precisa debugar. Opções possíveis:

  • 'off' — nunca.
  • 'on' — sempre (caro).
  • 'retain-on-failure' — só em falhas (sem retry).
  • 'on-first-retry' — este: balanço entre custo e debugging.

actionTimeout: 10_000 vs navigationTimeout: 15_000

Action (click, fill) é rápido — 10s é folga. Navegação pode envolver carregar página inteira — 15s dá margem. Timeouts separados permitem falhar rápido onde deveria ser rápido.

projects: [{ name: 'chromium' }]

Por enquanto só Chromium. Pra adicionar Firefox/WebKit: { name: 'firefox', use: { ...devices['Desktop Firefox'] } }. Playwright rodaria os 25 testes em cada browser. Útil pra regressão cross-browser — custa mais tempo, então só em PRs marcados ou em nightly.

4. Fixtures — test.extend<>()

Fixtures são o mecanismo do Playwright pra injetar dependências tipadas nos testes. Se você vem de pytest, é o conftest.py — com tipos. 

// fixtures/pages.ts
import { test as base } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
// ... outros imports

type Pages = {
  loginPage: LoginPage;
  dropdownPage: DropdownPage;
  checkboxesPage: CheckboxesPage;
  dynamicLoadingPage: DynamicLoadingPage;
  javascriptAlertsPage: JavaScriptAlertsPage;
};

export const test = base.extend<Pages>({
  loginPage: async ({ page }, use) => {
    await use(new LoginPage(page));
  },
  // ... idem pros outros 4
});

export { expect } from '@playwright/test';

Anatomia de uma fixture

loginPage: async ({ page }, use) => {
  // 1. SETUP: depende de `page` (outra fixture embutida do Playwright)
  const loginPage = new LoginPage(page);

  // 2. INJEÇÃO: passa o valor pro teste
  await use(loginPage);

  // 3. TEARDOWN: tudo que vier depois de `await use(...)` roda após o teste
  // (aqui não preciso de teardown — o BrowserContext é descartado sozinho)
}

Como um teste consome

// tests/login.spec.ts
import { test, expect } from '../fixtures/pages';
//                                 ^^^ importa DO fixture, não de @playwright/test

test('should log in', async ({ loginPage, page }) => {
  //                                    ^^^^^^^^^ ^^^^
  //                          fixture customizada fixture nativa
  await loginPage.visit();
  await loginPage.login('tomsmith', 'SuperSecretPassword!');
  await expect(page).toHaveURL(/\/secure/);
});
Diferença clássica — fixture vs beforeEach:
  • beforeEach: executa código antes do teste. Não retorna valor. Se precisar compartilhar dados, tem que usar variável no escopo da descrição — sem tipagem segura.
  • fixture: retorna um valor tipado, roda lazy (só se o teste declara), pode ter escopo worker (compartilhado entre testes do mesmo worker), e tem lifecycle setup+teardown.

5. Page Objects

Page Object Model (POM) separa o que a página tem (seletores) e o que faz (ações) em uma classe. Os testes viram narrativas de negócio, não scripts cheios de CSS selectors.

Exemplo anotado: LoginPage

import { Locator, Page } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly usernameInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly flashMessage: Locator;
  readonly logoutButton: Locator;

  constructor(page: Page) {
    this.page = page;
    this.usernameInput = page.locator('#username');
    this.passwordInput = page.locator('#password');
    this.submitButton = page.locator('button[type="submit"]');
    this.flashMessage = page.locator('#flash');
    this.logoutButton = page.locator('a[href="/logout"]');
  }

  async visit(): Promise<void> {
    await this.page.goto('/login');  // usa baseURL do config
  }

  async login(username: string, password: string): Promise<void> {
    await this.fillUsername(username);
    await this.fillPassword(password);
    await this.submit();
  }
  // ... métodos granulares (fillUsername, submit, logout)
}

Decisões importantes

EscolhaPor quê
readonly nas propriedades Impede reatribuição acidental (this.submitButton = ...). Os locators são ponteiros estáveis — nunca deveriam mudar.
Locators no construtor, não como strings Se o seletor mudar, muda em 1 lugar. Nos testes você usa loginPage.submitButton.click(), não page.locator('button[type="submit"]').click().
Métodos async retornando Promise<void> Tipo explícito. Se alguém esquecer o await, o TypeScript avisa (com no-floating-promises do ESLint, bloqueia).
Parâmetros tipados (username: string) Strict mode pega erros de chamada. Sem any implícito.
Método de alto nível (login) + granulares (fillUsername, submit) Testes felizes usam login(). Testes de borda (submit com campo vazio) usam só submit() direto. Granularidade pra flexibilidade.

Seletores — boas práticas

  • Prefira IDs e roles: #username, page.getByRole('button', { name: 'Login' }).
  • Evite classes CSS estilísticas: .btn-primary quebra quando redesign.
  • Use data-testid quando o time topar marcar os elementos: page.getByTestId('submit-btn') — estável como contrato.

6. Testes E2E

Exemplo: login.spec.ts

import { test, expect } from '../fixtures/pages';

test.describe('Login page', () => {
  test('should log in with valid credentials', async ({ loginPage, page }) => {
    await loginPage.visit();
    await loginPage.login('tomsmith', 'SuperSecretPassword!');

    await expect(page).toHaveURL(/\/secure/);
    await expect(loginPage.flashMessage).toContainText('You logged into a secure area!');
  });

  test('should display an error for an invalid username', async ({ loginPage }) => {
    await loginPage.visit();
    await loginPage.login('invaliduser', 'SuperSecretPassword!');

    await expect(loginPage.flashMessage).toContainText('Your username is invalid!');
  });
  // ... invalid password, empty fields, logout
});

Padrão AAA

Todos os testes seguem Arrange / Act / Assert:

  • Arrange: await loginPage.visit() — coloca a página no estado conhecido.
  • Act: await loginPage.login(...) — executa a ação sendo testada.
  • Assert: await expect(...) — verifica resultado.

Assertions importantes do Playwright

AssertionO que fazTem auto-wait?
toHaveURL(regex)Aguarda URL baterSim
toHaveText(...)Texto exatoSim
toContainText(...)Contém substringSim
toBeVisible()Elemento no DOM + visívelSim
toBeChecked()Checkbox/radio marcadoSim
toHaveCount(n)Número de elementos no locatorSim
Pegadinha: as assertions do Playwright expect(locator)... têm auto-wait. As assertions do Jest-style expect(valor).toBe(...) não — são síncronas. Se misturar (expect(await locator.textContent()).toBe(...)), você perde o retry automático.

Regra: pra DOM, sempre expect(locator).to...() com await.

7. API testing com request

Esse é um dos argumentos mais fortes pra escolher Playwright: API testing nativo, sem Postman/supertest/axios.

Como funciona

O request é uma fixture embutida. Quando você declara async ({ request }) => ..., Playwright te dá um cliente HTTP que:

  • Respeita baseURL e headers do config.
  • Pode compartilhar cookies/auth com o browser (via storageState).
  • Gera relatório no mesmo HTML report dos testes de UI.

Exemplos do api.spec.ts

GET simples com status

test('GET /status/200 returns 200 OK', async ({ request }) => {
  const response = await request.get('https://httpbin.org/status/200');

  expect(response.status()).toBe(200);
  expect(response.ok()).toBeTruthy();  // ok() é true pra 2xx
});

POST com body JSON

test('POST /post echoes the JSON payload', async ({ request }) => {
  const payload = { user: 'tomsmith', action: 'login' };

  const response = await request.post('https://httpbin.org/post', {
    data: payload,   // Playwright faz JSON.stringify e seta Content-Type automaticamente
  });

  expect(response.status()).toBe(200);
  const body = await response.json();
  expect(body.json).toEqual(payload);  // httpbin retorna o que mandei em body.json
});

Headers customizados (ex.: Authorization)

test('GET /headers echoes a custom Authorization header', async ({ request }) => {
  const response = await request.get('https://httpbin.org/headers', {
    headers: { Authorization: 'Bearer fake-token-123' },
  });

  const body = await response.json();
  expect(body.headers.Authorization).toBe('Bearer fake-token-123');
});

Query params

test('GET /get returns JSON with query params', async ({ request }) => {
  const response = await request.get('https://httpbin.org/get', {
    params: { q: 'playwright', lang: 'typescript' },
  });

  const body = await response.json();
  expect(body.args).toEqual({ q: 'playwright', lang: 'typescript' });
});

Segredos em testes de API

NUNCA commitar tokens/keys. SEMPRE:
  • .env local (ignorado no .gitignore) + .env.example sem valores reais.
  • GitHub Actions: secrets.API_TOKEN passa via env: no step.
  • Consumir no teste: process.env.API_TOKEN.
  • Em produção: usar usuário de serviço com escopo mínimo, nunca usuário real.
  • Dados PII: usar @faker-js/faker, nunca dados de cliente real.
  • Logs: mascarar headers Authorization antes de reportar.
// Exemplo do padrão recomendado
test('authenticated call', async ({ request }) => {
  const res = await request.get('/api/me', {
    headers: { Authorization: `Bearer ${process.env.API_TOKEN}` },
  });
  expect(res.status()).toBe(200);
});

8. Dialogs — alert, confirm, prompt

Dialogs nativos do browser (window.alert, window.confirm, window.prompt) travam a execução até serem respondidos. Playwright lida com eles via eventos.

Regra de ouro: registre ANTES de disparar

test('should accept a JS alert', async ({ javascriptAlertsPage, page }) => {
  await javascriptAlertsPage.visit();

  // 1. Registra o handler ANTES da ação que dispara o dialog
  page.once('dialog', (dialog) => dialog.accept());

  // 2. Agora dispara. Quando o alert abrir, o handler clica "OK".
  await javascriptAlertsPage.triggerAlert();

  await expect(javascriptAlertsPage.result).toHaveText('You successfully clicked an alert');
});

page.once vs page.on

  • page.once: handler executa 1 vez e desregistra. É o que você quer em testes.
  • page.on: handler persiste. Se houver outro dialog depois, ele ainda pega. Pode vazar entre testes se o contexto não for recriado.

Os 3 métodos de dialog

MétodoEquivalente UIUso
dialog.accept()Clicar "OK"Alert, confirm positivo
dialog.accept('texto')Preencher prompt e clicar "OK"Prompt aceito com valor
dialog.dismiss()Clicar "Cancelar" / XConfirm negativo, prompt cancelado

Os 5 testes em javascript-alerts.spec.ts

  1. Alert aceito — dialog.accept() → resultado "You successfully clicked an alert".
  2. Confirm aceito — dialog.accept() → "You clicked: Ok".
  3. Confirm cancelado — dialog.dismiss() → "You clicked: Cancel".
  4. Prompt com texto — dialog.accept('Hello from Playwright') → "You entered: Hello from Playwright".
  5. Prompt cancelado — dialog.dismiss() → "You entered: null".

9. CI/CD — GitHub Actions

.github/workflows/playwright.yml

name: Playwright Tests
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npx playwright install --with-deps chromium
      - run: npx playwright test
        env:
          CI: true

      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

Pontos de atenção

  • npm ci em vez de npm install: mais rápido, usa package-lock.json como verdade absoluta.
  • --with-deps chromium: instala os system deps do Linux que o Chromium precisa (Playwright cuida disso).
  • env: CI: true: ativa o ramo process.env.CI ? ... do config (retries, workers, forbidOnly).
  • if: always() no upload: faz upload do report mesmo quando testes falham — que é exatamente quando você mais quer o report.
  • retention-days: 30: artifact fica 30 dias disponível pra download no GitHub.

Como verificar o report

  1. Vai no PR → tab "Checks" → clica no run.
  2. Na tela do run, aba "Summary" → seção "Artifacts" → baixa playwright-report.zip.
  3. Descompacta e abre index.html no navegador.

import { Locator, Page } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly usernameInput: Locator;

  constructor(page: Page) {
    this.page = page;
    this.usernameInput = page.locator('#username');
  }

  async visit(): Promise<void> {
    await this.page.goto('/login');
  }

  async fillUsername(username: string): Promise<void> {
    await this.usernameInput.fill(username);
  }

Quando escolher TS vs Python pra QA? Depende do time. Se o produto é web/Node, TypeScript compartilha tipos com a aplicação (interfaces de API viram contratos de teste). Se o time é Python-heavy (data/ML), Python é menos fricção. Em stack Node, TypeScript reduz handoff e acelera onboarding.

11. Perguntas técnicas típicas

Playwright

locator vs page.$()

locator é lazy: só avalia quando age, tem auto-wait, retry automático. Se o DOM muda, re-avalia.

$ (API antiga, tipo Puppeteer) é snapshot: retorna ElementHandle no momento da chamada. Se o DOM re-renderiza, vira referência morta. Sempre prefira locator.

Por que evitar page.waitForTimeout(ms)

É sleep cego. Ou você espera demais (teste lento) ou de menos (flaky). Use expect(locator).toBeVisible() ou locator.waitFor(), que saem assim que a condição é verdade — com timeout configurável como limite superior.

Como mockar uma API

await page.route('**/api/users', (route) =>
  route.fulfill({ json: [{ id: 1, name: 'Mock User' }] })
);

Intercepta a requisição antes de sair. Útil pra testar UI sem depender de backend.

test.beforeEach vs fixture

beforeEach roda código; fixture injeta valor tipado, tem lifecycle setup+teardown, pode ter escopo worker. Fixtures são mais reutilizáveis e seguras.

Como rodar só testes de smoke

Tag + grep: test('@smoke logs in', ...) + npx playwright test --grep "@smoke".

QA em geral

Cypress vs Playwright

  • Cypress: roda dentro do browser (arquitetura in-browser). Só 1 browser por suíte. Time-travel debugging excelente. Comunidade enorme.
  • Playwright: controla o browser externamente (out-of-process). Multi-browser, multi-tab, multi-contexto (cenários com N usuários). API testing nativo.
  • Critério de escolha: stack do time, necessidade de multi-tab, e se vai testar API no mesmo projeto.
  • Resposta curta: "Cypress é excelente pra UI em 1 browser; Playwright escala melhor pra multi-contexto e cobre API no mesmo runner."

Manual vs automatizado

  • Automatiza: fluxos estáveis, críticos, repetitivos, baratos de manter.
  • Manual: exploratório, fluxos novos ainda instáveis, cenários visuais subjetivos, UX.
  • Critério: custo de automação + manutenção vs benefício de rodar N vezes.
  • Cuidado com afirmações absolutas como "tudo pode ser automatizado" — alguns testes custam mais caro pra manter que pra rodar manualmente.

Dados sensíveis em testes de API

Resposta detalhada na seção 7 (API testing) — resumo: nunca commitar tokens, usar .env + secrets de CI, usuário de serviço com escopo mínimo, mascarar headers em logs, dados fake via @faker-js/faker.

Pirâmide de testes

~70% unit (rápidos, devs) / ~20% integration / ~10% E2E (caros, lentos). QA tipicamente foca E2E + integration crítico quando os devs cobrem unit.

Linha a linha:

  • target: "ES2022" — o TS compila pra ES2022, que já tem async/await, top-level await, nullish coalescing etc. Node 20 executa ES2022 nativo.
  • module: "commonjs" — Playwright e a maioria das libs Node ainda usam require(). Poderia ser esnext, mas commonjs evita fricção de interop.
  • lib: ["ES2022", "DOM"] — habilita tipos do runtime JS + tipos do DOM (pra autocomplete de document, HTMLElement dentro de page.evaluate()).
  • strict: true — liga todas as flags estritas (noImplicitAny, strictNullChecks, strictFunctionTypes etc.). Decisão explícita: nunca aceitar any implícito.
  • esModuleInterop: true — permite import x from 'y' mesmo quando o módulo exporta via CommonJS. Sem isso, teria que usar import * as x from 'y' em vários pontos.
  • skipLibCheck: true — pula checagem de tipos de node_modules. Acelera compilação e evita erros vindos de libs de terceiros que você não controla.
  • forceConsistentCasingInFileNames: true — erro se você importar LoginPage vs loginpage. Mac é case-insensitive por padrão, Linux (CI) não — sem isso, o teste passa local e quebra no CI.
  • resolveJsonModule: true — permite import data from './fixtures/users.json'. Útil pra dados de teste sem precisar ler com fs.
  • moduleResolution: "node" — usa o algoritmo do Node pra resolver import (procura node_modules, package.json#main etc.).
  • baseUrl: "." + paths — habilita aliases. @pages/LoginPage resolve pra pages/LoginPage.ts. Evita ../../../pages/LoginPage em specs aninhados. OBS: o projeto não está usando os aliases no código atual (os imports são relativos), mas a config tá pronta pra quando escalar.
  • noEmit: true — o TS só verifica tipos; o Playwright Test compila o TS em runtime via ts-node/esbuild interno. Não gera arquivos .js.
  • include: ["**/*.ts"] — checa todo .ts do projeto.
  • exclude — ignora node_modules (dependências) e pastas geradas em runtime (test-results, playwright-report) pra não poluir a saída de type-check.

Decisão filosófica: strict: true é o que transforma TS em ferramenta de qualidade. Sem ele, TS vira "JS com anotações opcionais". Pergunta típica: "por que TS em QA?" — resposta: catches refactor bugs antes do teste rodar (errar o nome de um locator vira erro de compilação, não falha em runtime).

pages/DropdownPage.ts — dropdown <select> nativo 
import { Locator, Page } from '@playwright/test';

export class DropdownPage {
  readonly page: Page;
  readonly dropdown: Locator;

  constructor(page: Page) {
    this.page = page;
    this.dropdown = page.locator('#dropdown');
  }

  async visit(): Promise<void> {
    await this.page.goto('/dropdown');
  }

  async selectOption(value: string): Promise<void> {
    await this.dropdown.selectOption(value);
  }
}

Análise:

  • Seletor #dropdown — ID, estável. Match direto com o <select id="dropdown"> da página.
  • selectOption(value) — chama o método nativo do Playwright locator.selectOption(), que por baixo dos panos dispara o evento change correto. Nunca faça click em <option> — cada browser trata diferente.
  • Assinatura value: string porque o atributo HTML value="1" é string, mesmo parecendo número.
  • visit() usa path relativo /dropdown — o baseURL do playwright.config.ts é prefixado automaticamente.

Alternativas de selectOption:

  • selectOption('1') — por value (usado aqui).
  • selectOption({ label: 'Option 1' }) — por texto visível.
  • selectOption({ index: 1 }) — por posição.
  • selectOption(['1', '2']) — múltiplos em <select multiple>.
pages/CheckboxesPage.ts — coleção de checkboxes 
export class CheckboxesPage {
  readonly page: Page;
  readonly checkboxes: Locator;

  constructor(page: Page) {
    this.page = page;
    this.checkboxes = page.locator('#checkboxes input[type="checkbox"]');
  }

  async visit(): Promise<void> {
    await this.page.goto('/checkboxes');
  }

  getCheckbox(index: number): Locator {
    return this.checkboxes.nth(index);
  }

  async toggle(index: number): Promise<void> {
    await this.getCheckbox(index).click();
  }
}

Análise:

  • #checkboxes input[type="checkbox"] — resolve pra múltiplos elementos. O Locator representa a coleção, não um único. Playwright chama isso de "strict mode violation" se você tentar fazer click direto sem desambiguar.
  • getCheckbox(index) retorna um novo Locator restrito ao N-ésimo elemento. .nth(0) é o primeiro. Não executa query no momento da chamada — só guarda a descrição.
  • getCheckbox é síncrono e não tem Promise — só constrói o Locator. A query de verdade só acontece quando você chama .click(), .isChecked() etc.
  • toggle(index) — clica; o browser inverte o estado do checkbox nativamente. Não precisa de lógica condicional "se marcado desmarca senão marca".

Por que retornar Locator em vez de esconder? Porque o teste precisa asseverar sobre o estado: await expect(checkboxesPage.getCheckbox(0)).toBeChecked(). Se o método retornasse boolean, você teria dado um snapshot — Playwright quer Locators pra poder auto-esperar.

pages/DynamicLoadingPage.ts — elementos que aparecem depois 
export class DynamicLoadingPage {
  readonly page: Page;
  readonly startButton: Locator;
  readonly loadedText: Locator;
  readonly loadingIndicator: Locator;

  constructor(page: Page) {
    this.page = page;
    this.startButton = page.locator('#start button');
    this.loadedText = page.locator('#finish h4');
    this.loadingIndicator = page.locator('#loading');
  }

  async visitExample(exampleNumber: 1 | 2): Promise<void> {
    await this.page.goto(`/dynamic_loading/${exampleNumber}`);
  }

  async clickStart(): Promise<void> {
    await this.startButton.click();
  }
}

Análise:

  • Tipo literal 1 | 2 — union type. O TS impede visitExample(3) em compile-time. É uma mini-enum inline. Se a página tivesse 5 exemplos, 1 | 2 | 3 | 4 | 5 ou um enum próprio.
  • Template literal `/dynamic_loading/${exampleNumber}` — interpolação nativa do TS/JS. Diferente do Python (f"...") mas mesma ideia.
  • loadingIndicator está declarado mas não é usado nos testes atuais. É intencional: fica disponível pro caso de você querer asseverar "o spinner aparece antes e some depois" num teste mais rigoroso.
  • #finish h4 — seletor combinado (descendente direto ou indireto). Match num <div id="finish"><h4>Hello World!</h4></div>.
  • Não tem método waitForLoad(). Por que? Porque no teste a gente usa await expect(loadedText).toBeVisible({ timeout: 10_000 }) — a assertion já espera. Não precisa de método customizado.

Lição: o POM expõe o que a página tem e o que o usuário faz. A espera é responsabilidade da assertion, não do POM.

pages/JavaScriptAlertsPage.ts — três botões, três dialogs 
export class JavaScriptAlertsPage {
  readonly alertButton: Locator;
  readonly confirmButton: Locator;
  readonly promptButton: Locator;
  readonly result: Locator;

  constructor(page: Page) {
    this.page = page;
    this.alertButton = page.getByRole('button', { name: 'Click for JS Alert' });
    this.confirmButton = page.getByRole('button', { name: 'Click for JS Confirm' });
    this.promptButton = page.getByRole('button', { name: 'Click for JS Prompt' });
    this.result = page.locator('#result');
  }

  async triggerAlert(): Promise<void> {
    await this.alertButton.click();
  }

  async triggerConfirm(): Promise<void> { await this.confirmButton.click(); }
  async triggerPrompt(): Promise<void>  { await this.promptButton.click(); }
}

Análise:

  • Mudança de estratégia de seletor: aqui os botões usam page.getByRole('button', { name: '...' }) em vez de CSS. Por quê? Porque essa é a forma recomendada pelo Playwright — é a mesma heurística que um usuário de screen reader usa. Mais resiliente a mudanças de ID/classe e alinhada com a11y.
  • #result continua CSS porque é um container de display sem role semântica — ID é o seletor certo.
  • Nenhum método aqui registra o dialog handler. Decisão crítica: o registro fica na spec, não no POM. Motivo: cada teste quer um comportamento diferente (accept / dismiss / accept com texto). Colocar no POM forçaria variantes (triggerAlertAndAccept, triggerAlertAndDismiss) — mais código, menos flexibilidade.
  • JSDoc no topo do arquivo avisa sobre isso: "Note: the dialog handler must be registered before the click..."

Padrão que emerge: POM expõe o disparador (clicar no botão); a spec orquestra o contexto (que listener registrar).

tests/dropdown.spec.ts — 4 testes 
import { test, expect } from '../fixtures/pages';

test.describe('Dropdown page', () => {
  test('should have no option selected by default', async ({ dropdownPage }) => {
    await dropdownPage.visit();
    await expect(dropdownPage.dropdown).toHaveValue('');
  });

  test('should select option 1', async ({ dropdownPage }) => {
    await dropdownPage.visit();
    await dropdownPage.selectOption('1');
    await expect(dropdownPage.dropdown).toHaveValue('1');
  });

  // ... (option 2, switch 1→2)
});

Pontos didáticos:

  • Import de ../fixtures/pages, não de @playwright/test — garante que dropdownPage seja injetado.
  • toHaveValue('') cobre o estado inicial — o <option> placeholder tem value="". Asseverar string vazia é mais preciso do que "não tem valor".
  • O 4º teste (switch) é o mais valioso: simula interação sequencial, pegando bugs onde "selecionar 1 bloqueia selecionar 2" (já vi em combobox com estado interno bugado).
  • Isolamento: cada teste começa com visit(). Não há beforeEach. Decisão: clareza > DRY pra testes curtos. Em suites grandes, beforeEach faz sentido.
tests/checkboxes.spec.ts — 4 testes 
test('should have correct initial states', async ({ checkboxesPage }) => {
  await checkboxesPage.visit();
  await expect(checkboxesPage.getCheckbox(0)).not.toBeChecked();
  await expect(checkboxesPage.getCheckbox(1)).toBeChecked();
});

test('should toggle both checkboxes', async ({ checkboxesPage }) => {
  await checkboxesPage.visit();
  await checkboxesPage.toggle(0);
  await checkboxesPage.toggle(1);
  await expect(checkboxesPage.getCheckbox(0)).toBeChecked();
  await expect(checkboxesPage.getCheckbox(1)).not.toBeChecked();
});

Análise:

  • Estado inicial: o HTML real do site tem checkbox 1 desmarcado e checkbox 2 marcado (bizarro de propósito pra testar). O primeiro teste documenta esse estado e detecta se o site mudar.
  • .not.toBeChecked() — negação encadeada. Note que no Python é not_to_be_checked() como método; em TS é .not. como proxy.
  • toggle(0) + toggle(1) — cobre o caso "usuário interage com os dois em sequência". Importante porque um bug comum é "toggle afeta o outro por erro de listener".
  • Cada assertion é await expect(...). Esquecer o await é o erro #1 em TS: a assertion vira um Promise pendente que o test runner nunca vê, o teste "passa" mas não valida nada.
tests/dynamic-loading.spec.ts — 2 testes, auto-wait 
test('should reveal a hidden element after loading (example 1)', async ({ dynamicLoadingPage }) => {
  await dynamicLoadingPage.visitExample(1);
  await dynamicLoadingPage.clickStart();
  await expect(dynamicLoadingPage.loadedText).toBeVisible({ timeout: 10_000 });
  await expect(dynamicLoadingPage.loadedText).toHaveText('Hello World!');
});

test('should render an element that was not in the DOM (example 2)', async ({ dynamicLoadingPage }) => {
  await dynamicLoadingPage.visitExample(2);
  await dynamicLoadingPage.clickStart();
  await expect(dynamicLoadingPage.loadedText).toBeVisible({ timeout: 10_000 });
  await expect(dynamicLoadingPage.loadedText).toHaveText('Hello World!');
});

Por que dois testes quase idênticos?

  • Exemplo 1: elemento existe no DOM desde o início, com display: none. Aparece depois.
  • Exemplo 2: elemento não existe no DOM inicial. É injetado depois do fetch.
  • Os dois cenários estressam mecanismos diferentes de auto-wait do Playwright. No exemplo 1, ele precisa esperar display mudar; no 2, esperar o elemento existir. Se o Playwright falhasse num desses, o bug ficaria escondido.

{ timeout: 10_000 } — separador numérico do ES2021, só legibilidade (10_000 === 10000). Override local do timeout global (5s). O site simula 5s de latência, então 10s dá margem.

Por que toBeVisible() antes de toHaveText()?

  • Tecnicamente, toHaveText() já faz auto-wait. Poderia ser só a segunda assertion.
  • Mas separar comunica intenção: "primeiro eu provo que apareceu, depois que o texto tá certo". Se falhar em toBeVisible, você sabe que é problema de rendering; se falhar em toHaveText, é conteúdo.
  • Trade-off: duas assertions são mais lentas (cada uma tem seu poll interval). Em suites enormes, consolidar faz sentido.
tests/javascript-alerts.spec.ts — 5 testes, 3 dialogs 
test('should accept a JS alert', async ({ javascriptAlertsPage, page }) => {
  await javascriptAlertsPage.visit();
  page.once('dialog', (dialog) => dialog.accept());     // ← ANTES do click
  await javascriptAlertsPage.triggerAlert();
  await expect(javascriptAlertsPage.result).toHaveText('You successfully clicked an alert');
});

test('should enter text in a JS prompt and accept it', async ({ javascriptAlertsPage, page }) => {
  await javascriptAlertsPage.visit();
  page.once('dialog', (dialog) => dialog.accept('Hello from Playwright'));
  await javascriptAlertsPage.triggerPrompt();
  await expect(javascriptAlertsPage.result).toHaveText('You entered: Hello from Playwright');
});

test('should dismiss a JS prompt without text', async ({ javascriptAlertsPage, page }) => {
  await javascriptAlertsPage.visit();
  page.once('dialog', (dialog) => dialog.dismiss());
  await javascriptAlertsPage.triggerPrompt();
  await expect(javascriptAlertsPage.result).toHaveText('You entered: null');
});

Destrava cada linha:

  • Por que { javascriptAlertsPage, page }? O teste precisa dos dois: a page object pros botões, e page cru pra registrar o listener. Fixtures podem ser injetadas lado a lado.
  • page.once vs page.on: once remove o listener depois de disparar uma vez. Se um teste clica duas vezes sem querer (bug), on aceitaria os dois; once deixaria o segundo travar a página — melhor pra detectar regressões.
  • Teste 5 ("dismiss a JS prompt without text"): o resultado é "You entered: null" (string literal "null" no DOM). Sutileza: o JS do site concatena o retorno do prompt(), e dismissar retorna null, que vira "null" ao concatenar com string. Asseverar esse exato texto pega bugs onde alguém "consertar" o comportamento pra mostrar string vazia.
  • Teste 4 (prompt com texto): dialog.accept('Hello from Playwright') — o texto passado substitui o que o usuário digitaria. O site ecoa em #result.

Padrão arquitetural: a sequência é sempre:

  1. Navegar (.visit()).
  2. Registrar listener (page.once('dialog', ...)).
  3. Disparar ação (.triggerXxx()).
  4. Asseverar resultado (expect(result).toHaveText(...)).

Inverter 2 e 3 (listener depois do click) é o bug clássico. Playwright não bufferiza dialogs — se ninguém tá ouvindo, o dialog trava a página e o teste dá timeout.
